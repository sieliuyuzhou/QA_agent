# Vibe Coding 实战：用 AI 搭建 RAG 检索增强服务

## 第一章：项目规则

### 1.1 我们要做什么

一个从本地文档入库到 HTTP 检索的完整 RAG 服务。用户将 FAQ 文档向量化存入数据库，通过 HTTP 接口发起检索请求，系统自动改写查询并返回最相关的问答对。

### 1.2 分层架构

```
应用层（api）         ← 对外暴露接口
     ↓
领域层（retriever）    ← 编排业务流程
     ↓
基础层（model / rag）  ← 提供原子能力
```

依赖只允许往下指：上层依赖下层，下层不知道上层存在。同层模块互不依赖。

### 1.3 模块边界三原则

1. **单向依赖**：依赖关系只能从上往下，禁止反向依赖或循环依赖
2. **模块公开接口隔离**：模块间只通过 `__init__.py` 的导出来交互，禁止 import 其他模块的内部文件
3. **配置隔离**：每个模块用独立前缀的环境变量（如 `MODEL_*` / `EMBEDDING_*` / `RETRIEVER_*` / `API_*`），避免冲突

### 1.4 开发建议

- 功能最小可用即可，不做过度设计
- 增量式开发，先跑通再迭代
- 新依赖写入 requirements.txt
- 每完成一个模块就验证，不要堆到最后

### 1.5 配置管理

项目使用一份 `.env` 管理所有配置，不需要每个模块单独维护配置文件。每个模块将配置信息作为类的属性或构造函数参数，带合理默认值，通过模块级单例对外暴露。如果多个模块使用相同的 API 密钥或端点，在 `.env` 中用不同前缀指向同一个值即可，不需要代码层面去统一。

## 第二章：基础层 —— Model 模型调用模块

统一的文本生成调用模块，配置信息用属性直接赋值，方法封装在同一个类下，对外暴露模块级单例。提供同步与流式对话、提示词模板、重试与异常处理等能力。本模块只负责文本生成，不涉及 Embedding。

**完成标志**：能调用大模型并获得同步和流式响应，异常场景（超时、限流）能正确重试或抛出对应异常。

**AI 协作提示**：接口兼容格式（如 OpenAI SDK 格式），以及你需要的能力范围（同步/流式/模板/重试/异常分层）。

---

## 第三章：基础层 —— RAG 向量存储模块

将本地 FAQ 文档向量化并存入向量数据库，对外提供写入和检索能力。本模块只负责"存"和"查"，不涉及文本生成。Embedding 作为模块内的独立实体，拥有自己的类，配置用属性直接赋值，自行对接向量化接口。

**完成标志**：能将 FAQ 文档入库并检索到相关结果，写入条数与文档中的 Q&A 对数量一致。

**AI 协作提示**：开始前告诉 AI 你的数据源格式（比如 FAQ 文档的结构）、向量数据库的选型、Embedding 接口的兼容格式。Embedding 是 RAG 内部的独立实体，不需要依赖 model 模块。

**常见偏差**：把文本生成逻辑放进 RAG 模块。RAG 只做向量的存和查，任何涉及文本生成的操作都不属于这里。

---

## 第四章：领域层 —— Retriever 检索增强模块

编排层，组合 Model 的查询改写能力与 RAG 的向量检索能力。不引入新依赖，只通过已有模块的模块公开接口（`__init__.py` 导出）进行调用。

**完成标志**：用户问题经过改写后检索，结果比直接检索更精准（对比改写前后的检索距离）。

**AI 协作提示**：开始前告诉 AI 你已经完成了 model 和 rag 两个模块，说明它们各自的公开接口和用法，让 AI 基于现有模块来编排，而不是从零设计。

**常见偏差**：绕过模块公开接口，直接 import model 或 rag 的内部文件（如 `from model.chat import ChatService`）。必须通过 `__init__.py` 的导出来调用。

---

## 第五章：应用层 —— API 接口服务

对外暴露统一的 HTTP 检索接口，通过参数控制是否启用查询改写。只依赖 Retriever 模块，不直接访问下层。

**完成标志**：启动服务后，通过 HTTP 请求能完成一次完整检索（改写 + 向量查询 + 返回结果）。

**AI 协作提示**：开始前告诉 AI 你已经有 retriever 模块及其 pipeline 的用法，API 层只负责参数接收、调用 pipeline、格式化响应，不涉及任何业务逻辑。

**常见偏差**：在 API 层直接 import rag 或 model 的内容。API 只依赖 retriever，其他模块对 API 是不可见的。

## 第六章：方法论 —— 与 AI 协作解决问题

### 6.1 描述问题，而非猜测原因

告诉 AI 你观察到的现象（"检索返回空"），而不是你的诊断（"数据库路径错了"）。让 AI 帮你分析调用链路，它的追踪能力比你手动排查强得多。

### 6.2 让 AI 追踪调用链

当结果不符合预期，要求 AI 从入口到终点逐层追踪，定位断裂点。比如："请分析从 HTTP 请求到检索结果的完整调用链路，找出数据在哪里丢失了。"

### 6.3 先设计方案，再动手执行

要求 AI 先列出设计方案或任务清单，确认思路无误后再执行。无论是新功能开发还是 Bug 修复，都应该让 AI 先给出思路，你确认方向正确后再让它动手。避免 AI 在不了解全貌时盲目行动。

### 6.4 关注边界条件

相对路径、CWD 依赖、环境变量缺失 —— 这些是 Vibe Coding 中最常见的隐性 Bug。AI 生成的代码默认使用相对路径，不会主动考虑"从不同目录启动"的场景。当你发现行为与预期不一致时，优先检查路径和环境变量。

### 6.5 增量验证

每完成一个模块就跑验证脚本，不要堆到最后才发现问题。一个问题在它诞生的模块里排查，比在整个项目里排查简单十倍。

### A. 模块依赖图（构建顺序）

```
┌─────────────┐
│     api      │  ← 第 4 个构建
└──────┬──────┘
       ↓
┌─────────────┐
│   retriever  │  ← 第 3 个构建
└──┬───────┬──┘
   ↓       ↓
┌───────┐ ┌──────┐
│ model  │ │  rag  │  ← 第 1、2 个构建（顺序不限，互不依赖）
└───────┘ └──────┘
```

构建顺序：先完成基础层（model / rag，顺序不限），再完成领域层（retriever），最后完成应用层（api）。

### B. 数据流图（一个请求的完整路径）

```
用户请求
  │
  │  POST /search  {"question": "怎么退款", "enable_rewrite": true}
  ↓
┌────────── api ──────────┐
│  routes.py              │
│    → 接收请求参数        │
│    → 调用 pipeline      │
│    → 格式化响应          │
└──────────┬──────────────┘
           ↓
┌──────── retriever ──────────────┐
│  pipeline.py                    │
│    → 决定是否改写                │
│    → 改写 → 检索                │
│                                  │
│  ┌── rewriter.py ──┐            │
│  │  调用 model 改写  │──→ model（文本生成）
│  └─────────────────┘            │
│  ┌── pipeline.py ──┐            │
│  │  调用 rag 检索    │──→ rag（Embedding + 向量检索）
│  └─────────────────┘            │
└──────────┬──────────────────────┘
           ↓
┌──── 响应 ───────────────┐
│  {                       │
│    "original_question":  │
│    "rewritten_query":    │
│    "results": [...]      │
│  }                       │
└──────────────────────────┘
```
